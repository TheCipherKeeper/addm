# Сопровождение методологии

Этот документ предназначен человеку, который меняет саму методологию. Правила
ветвления и публикации задаёт корневой `AGENTS.md`; здесь описана область влияния
изменения и выпуск новой версии.

## Найти владельца факта

1. Роли и инварианты принадлежат `START.md`.
2. Жизненный цикл задачи принадлежит `WORKFLOW.md`.
3. Архитектурные ограничения принадлежат `ARCHITECTURE.md`.
4. Поставка и свидетельства принадлежат `OPERATIONS.md`.
5. Допустимые значения и форматы принадлежат `REFERENCE.md` и JSON-схемам.
6. Корневой `AGENTS.md` содержит только правила работы в этом репозитории.

Напоминание ссылается на каноническое правило по идентификатору и не пересказывает
его. Карта [`COVERAGE.md`](COVERAGE.md) связывает правило с автоматической проверкой.

## Классифицировать изменение

Несовместимым считается изменение, которое требует менять существующий
`BACKLOG.md`, `.tasks`, `.evidence`, `.methodology.yml`, структуру компонента или
настройки обязательного CI. Оно получает `BREAKING CHANGE:` в коммите и раздел
миграции в описании PR.

Изменение сообщения проверки, пояснения или необязательной рекомендации совместимо,
если допустимое множество репозиториев и действий не меняется.

## Проверить область влияния

- изменение документа — проверить ссылки, идентификаторы правил и отсутствие второго канона;
- изменение схемы — обновить генератор, пример, проверки и тест миграции;
- изменение скелета — синхронизировать производные файлы;
- изменение Python-инструмента — обновить модульные и сквозные тесты;
- изменение workflow — проверить все четыре заготовки и порядок стадий.

Перед коммитом выполняются:

```bash
uv run tools/skeletons/sync.py --check
uv run tools/verify/verify.py --report verification.json
uv run --with pytest==9.1.1 pytest
```

## Выпустить и обновить потребителей

После слияния отдельная задача создаёт стабильный тег. Для несовместимой версии
описание выпуска перечисляет изменённые форматы, команду преобразования и порядок
обновления закреплённого `methodology_ref`. Потребитель сначала применяет миграцию
в рабочей ветке, затем меняет закреплённую версию и проходит новый гейт.

Текущая машинная модель задач и свидетельств имеет `schema_version: 2`. Миграция
с версии 1 выполняется до изменения закреплённого `methodology_ref`:

```bash
uv run tools/migrate/v1_to_v2.py --root <hub> --repository-id <id> \
  --ci-url TASK-0001=https://github.com/owner/repo/actions/runs/123
```

`--ci-url` повторяется для каждого существующего evidence: ссылка становится
проверяемой аттестацией поставки и заменяет старые непрозрачные `source`. Команда
заменяет `target` на `targets`, группирует поля в `deliveries`, добавляет
`repository_id` и пересоздаёт `.tasks`. Evidence без независимой проверки не
мигрируется автоматически: сначала требуется восстановить подтверждение.
