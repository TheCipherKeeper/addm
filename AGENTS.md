# AGENTS.md — правила работы в репозитории

Этот файл — для людей и AI-агентов, которые работают с кодом. Здесь только
правила. Архитектура — в `docs/ARCHITECTURE.md`, раскладка — в `docs/LAYOUT.md`,
деплой — в `docs/DEPLOYMENT.md`, контракты модулей — в `docs/specs/`, детали
стека — в `docs/STACKS.md`.

> Это шаблон системы микросервисов в одном монорепо. Программа = совокупность
> сервисов; сервисы общаются через **брокер** (один на систему); деплой —
> контейнеры в **docker compose**. Каждый сервис реализуется **на одном из
> стеков**: Python, Go, Rust, TypeScript (бэкенд). Правило: один сервис —
> один язык; разные сервисы могут быть на разных стеках. Шаблон стека — в
> `docs/STACKS.md`, дерево каталогов — в `docs/LAYOUT.md`.

## Документация

Документы в порядке убывания приоритета:

1. Внешний хаб (если есть) — системные документы (состав, ADR).
2. `AGENTS.md` (этот файл) — правила работы.
3. `docs/ARCHITECTURE.md` — архитектура системы.
4. `docs/BACKLOG.md` — очередь задач.
5. `docs/specs/**/*.md` — контракты модулей.
6. Код — реализация.

`docs/INDEX.md` — карта по всем документам. Начни с него.

Если документы противоречат друг другу — побеждает старший. Расхождение —
повод для ADR.

## Модель ветвления

```
main ← merge только из dev
dev  ← PR из feature-веток
feat/<задача> ← от dev, удаляется после merge
```

- `main` — стабильная. Вливается только из `dev`.
- `dev` — разработческая. Вливается из feature-веток через PR.
- Прямой коммит в `main` или `dev` — запрещён. Только через feature-ветку + PR.

## Рабочий цикл

### Важно!
Бэклог — это строгая последовательная очередь (как миграции). Запрещено выполнять задачи параллельно или перепрыгивать через невыполненный пункт. Задача $N+1$ может быть начата только после полного завершения задачи $N$.

1. Прочитать `docs/BACKLOG.md` — взять самый первый невыполненный `[ ]` пункт.
   Тег задачи указывает сервис (и модуль), которого касается задача.
2. Прочитать `docs/INDEX.md` → `docs/ARCHITECTURE.md` → спек нужного модуля
   (возможно нескольких необходимых).
3. Создать ветку от dev: `git checkout dev && git pull && git checkout -b feat/<задача>`.
4. Изменить код. Написать тесты по пунктам «Что есть» в спеке.
5. Проверить — все три группы команд должны пройти (команды под стек сервиса —
   из таблицы ниже): **lint**, **test**, **build**. Если задача затрагивает
   несколько сервисов — проверить каждый затронутый. Если любая fail — чинить, повторить.
6. Обновить спек: перенести реализованное из «Что TODO» в «Что есть».
7. Обновить BACKLOG: поставить `[x]` на выполненный пункт.
8. Закоммитить (Conventional Commits, см. ниже).
9. Запушить, открыть PR в `dev`.
10. После merge — ветка удаляется.

## Команды проверки по стеку

Стек выбирается **per-сервис** и закрепляется один раз. Подставь набор для
каждого сервиса и держи единым. Полная конфигурация toolchain'а — в `docs/STACKS.md`.

| Стек | lint | test | build |
|---|---|---|---|
| **Python** | `ruff format --check . && ruff check .` | `pytest` | `uv build` |
| **Go** | `gofmt -l . && go vet ./...` | `go test ./...` | `go build -o bin/<service> ./cmd/<service>` |
| **Rust** | `cargo fmt --check && cargo clippy -- -D warnings` | `cargo test` | `cargo build --release` |
| **TypeScript** | `pnpm lint && tsc --noEmit` | `pnpm test` | `pnpm build` |

> Go: `gofmt -l .` должен вывести пустоту — это и есть «fmt --check».
> Python: опционально `mypy`/`pyright` для типизации.
> TypeScript: runtime — Node (по умолчанию); Deno/Bun допустимы, но зафиксируй в `docs/STACKS.md`.

## Интеграция сервисов

- Сервисы общаются **только через брокер** (Kafka/Redpanda/NATS — один на систему).
  Прямых HTTP-вызовов между сервисами избегать; если нужны — обосновать ADR.
- Топики/предметные области и формат сообщений (event envelope) описываются в
  `docs/ARCHITECTURE.md` (раздел «Брокер») и/или `docs/CONVENTIONS.md`.
- Контракты общих схем живут в `shared/` (если есть) и/или в спеках.

## Деплой

- Локальная разработка и деплой — через `docker-compose.yml` (см. `docs/DEPLOYMENT.md`).
- Каждый сервис имеет свой `Dockerfile` в `services/<service>/`.
- Брокер и общая инфраструктура (сеть, volumes) описываются в корневом
  `docker-compose.yml`, не в сервисах.
- Сборка контейнеров — в рамках команд `build` стека; образ собирается из `Dockerfile`.

## Что можно

- Писать код в `services/<service>/` и (опц.) `shared/`.
- Менять конфигурацию сборки/манифесты сервисов с обоснованием.
- Менять `docker-compose.yml`, `Dockerfile` сервисов, `.env.example` с обоснованием.
- Обновлять `docs/`.
- Создавать feature-ветки, коммитить, пушить, открывать PR в `dev`.
- Обновлять спеки после реализации.
- Заводить новые сервисы в `services/` (с picksом стека, Dockerfile, спекой).

## Что нельзя

- Коммитить напрямую в `main` или `dev`.
- Смешивать стеки внутри одного сервиса (один сервис — один язык).
- Вводить прямую связность между сервисами в обход брокера без ADR.
- Создавать ADR вне отведённого места (см. *ADR* ниже).
- Добавлять зависимости (включая образы в compose) без обоснования.
- Выдавать stub за реальную реализацию. Честно помечать: что placeholder, что TODO.
- Трогать lock-файлы (`Cargo.lock`, `go.sum`, `pnpm-lock.yaml`, `uv.lock`),
  конфиги окружения (`.env`), артефакты сборки без одобрения.

## Коммиты

Conventional Commits. Полностью на английском (или на языке проекта — задай в *Язык*).

```
feat(collector): add zfs-snapshot fs probe
fix(broker-gw): reject path traversal in HostBridge
docs: update ARCHITECTURE.md with service matrix
refactor(collector): extract envelope signing
chore(deploy): pin redpanda image in compose
```

Scope — имя сервиса или `deploy`/`docs`/`chore`. Breaking changes —
`BREAKING CHANGE:` в теле.

## ADR

Архитектурные решения фиксируются в ADR. По умолчанию ADR живут в
`docs/adr/` этого репозитория (формат — в `docs/adr/0001-record-architecture-decisions.md`).

> **Заметка про хаб.** Если проект — часть композиции репозиториев с общим
> хабом, выноси ADR в хаб (например `cybercity/adr/`), а из
> `docs/ARCHITECTURE.md` ссылайся на них. В таком случае удали `docs/adr/`
> из этого репо и убери пункт «создавать ADR в `docs/adr/`» из *Что нельзя*.
> Старые ADR помечай `Superseded`, не удаляй.
